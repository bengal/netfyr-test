# SPEC-503: YAML Reference Man Page

## What

Write a `netfyr.yaml(5)` man page that documents the YAML file formats used by netfyr. This is a hand-written man page in troff format (not auto-generated from clap) that covers:

1. **Bare state shorthand** — the simplest format, with `type`, selector properties, and config fields all at the top level.
2. **Explicit policy format** — `kind: policy` wrapper with `name`, `factory`, `priority`, `selector`, `state`/`states`.
3. **Multi-document files** — using `---` separators.

For each format the man page documents every field, its type, whether it is required or optional, its default value (if any), and at least one inline example. The man page also documents the `Value` type mapping (how YAML scalars map to netfyr types), selector fields, and supported factory types.

## Why

The man(1) pages generated by clap_mangen (SPEC-501) document the CLI flags and arguments but say nothing about the YAML file formats that users actually write. A section 5 man page (`netfyr.yaml(5)`) is the Unix-standard way to document file formats. Having a comprehensive YAML reference available offline via `man netfyr.yaml` is essential for users configuring servers without internet access or who prefer terminal-based documentation.

## User interaction

```bash
# View the YAML reference
man netfyr.yaml

# Or with an explicit section
man 5 netfyr.yaml
```

### Man page structure

```
NETFYR.YAML(5)             File Formats Manual             NETFYR.YAML(5)

NAME
       netfyr.yaml — netfyr policy file formats

DESCRIPTION
       netfyr reads YAML files from /etc/netfyr/policies/ to configure
       network interfaces. This page documents all supported YAML formats.

       Files may contain multiple YAML documents separated by "---".
       Each document is parsed independently.

BARE STATE FORMAT
       The simplest format. A YAML document with no kind field declares
       the desired state of a single network entity. It is automatically
       wrapped into a static policy with default priority 100 and a name
       derived from the filename.

       All properties are written at the top level in a flat format:
       type and selector properties (name, driver, mac, pci_path) identify
       the target entity, and configuration properties (mtu, addresses,
       etc.) declare its desired state.

       Fields:
           type (string, required)
               The entity type. Currently supported: "ethernet".

           Selector properties (at top level, see SELECTORS below):
               name, driver, mac, pci_path

           Configuration properties (at top level, see FIELDS below):
               mtu, addresses, routes, state, etc.

       Example:
           type: ethernet
           name: eth0
           mtu: 1500
           addresses:
             - 10.0.1.50/24

POLICY FORMAT
       A YAML document with kind: policy. Provides explicit control over
       the factory type, priority, and naming.

       Fields:
           kind (string, required)
               Must be "policy".

           name (string, required)
               Unique policy name. Used in conflict reports and
               provenance tracking.

           factory (string, required)
               Factory type. One of: "static", "dhcpv4".

           priority (integer, optional, default: 100)
               Numeric priority for field-level conflict resolution.
               Higher values take precedence.

           selector (mapping, required for dhcpv4)
               Identifies the target interface for DHCP. For static
               policies, the selector is inside state/states instead.

           state (mapping, optional)
               A single entity state definition in flat format (same
               structure as bare state format). Mutually exclusive
               with states. Used with factory: static.

           states (sequence, optional)
               A list of entity state definitions in flat format. Used
               when a policy produces multiple entities. Mutually
               exclusive with state. Used with factory: static.

       Example (static, single state):
           kind: policy
           name: server-network
           factory: static
           priority: 150
           state:
             type: ethernet
             name: eth0
             mtu: 9000
             addresses:
               - 10.0.1.50/24

       Example (static, multiple states):
           kind: policy
           name: server-network
           factory: static
           priority: 150
           states:
             - type: ethernet
               name: eth0
               mtu: 9000
               addresses:
                 - 10.0.1.50/24

       Example (dhcpv4):
           kind: policy
           name: eth0-dhcp
           factory: dhcpv4
           selector:
             name: eth0

MULTI-DOCUMENT FILES
       A single YAML file may contain multiple documents separated by
       "---". Each document is parsed independently and produces one
       policy.

       Example:
           type: ethernet
           name: eth0
           mtu: 9000
           ---
           type: ethernet
           name: eth1
           mtu: 1500

SELECTORS
       Selector fields identify which system entity a state targets.
       They are written at the top level of a state definition,
       alongside the type and configuration fields. All specified
       selector fields must match (AND logic).

           name (string)
               Exact interface name (e.g., "eth0").

           driver (string)
               Kernel driver name (e.g., "ixgbe").

           pci_path (string)
               PCI device path (e.g., "0000:03:00.0").

           mac (string)
               MAC address in colon-separated hex (e.g.,
               "aa:bb:cc:dd:ee:ff"). Comparison is case-insensitive.

       Note: In dhcpv4 policies, selector fields are wrapped in a
       selector: mapping at the policy level (not inside a state
       block), because the policy-level name field would collide
       with the interface name selector.

FIELDS
       Configuration fields are written at the top level of a state
       definition, alongside the type and selector properties. The
       available fields depend on the entity type.

       Ethernet fields:
           mtu (integer)
               Maximum transmission unit.

           addresses (sequence of strings)
               IPv4 addresses in CIDR notation (e.g., "10.0.1.50/24").
               Order is preserved: the first address becomes the primary
               (source) address. Duplicate addresses are rejected.
               IPv6 is not supported.

           routes (sequence of mappings)
               Each route has: destination (CIDR string), gateway (IP
               string), and optionally metric (integer).

           state (string)
               Administrative state: "up" or "down".

VALUE TYPES
       YAML scalars are mapped to netfyr types as follows:

           YAML boolean           netfyr Bool
           YAML integer >= 0      netfyr U64
           YAML integer < 0       netfyr I64
           YAML string "a.b.c.d"  netfyr IpAddr (if valid IPv4)
           YAML string "a.b/n"    netfyr IpNetwork (if valid IPv4 CIDR)
           YAML string (other)    netfyr String
           YAML sequence          netfyr List
           YAML mapping           netfyr Map

       IPv6 addresses are not supported. Strings containing IPv6
       addresses (e.g., "fe80::1") are treated as plain strings and
       will fail schema validation if used in address fields.

FILES
       /etc/netfyr/policies/
              Policy and bare state YAML files.

       /var/lib/netfyr/policies/
              Daemon-persisted policies (managed automatically).

SEE ALSO
       netfyr(1), netfyr-apply(1), netfyr-query(1), netfyr-history(1),
       netfyr-revert(1), netfyr-daemon(8), netfyr-examples(7)
```
